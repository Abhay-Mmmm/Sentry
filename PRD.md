# PRD.md — Project Sentinel: Local System Anomaly Monitor

---

## 1. Executive Summary

Project Sentinel is an entirely offline, privacy-first system anomaly detection and explanation platform designed to run on a single developer workstation. It ingests simulated process-level telemetry (CPU, memory, network, disk I/O), applies an unsupervised machine learning model to surface statistical outliers in real time, and feeds those anomalies to a locally hosted large language model to generate human-readable security narratives — all without a single byte leaving the machine.

The system is purpose-built for a 60-minute parallel-team sprint. Every architectural decision — from the monolithic orchestrator to the in-memory data layer — directly reflects this constraint. The goal is a fully demonstrable, end-to-end anomaly pipeline that runs from `streamlit run app.py` with zero external dependencies, zero cloud calls, and zero configuration drift across a 6-person team.

---

## 2. The 60-Minute Scope Compromises

These are deliberate, documented engineering trade-offs, not shortcuts. Each compromise was chosen to eliminate an entire class of integration risk within the sprint window.

### 2.1 Synthetic Data Loops Instead of Live Kernel Telemetry

**Decision:** Process telemetry is generated by a Python function using `numpy` random distributions seeded with realistic operating ranges. No eBPF probes, no `psutil` polling loops, no `/proc` filesystem reads.

**Justification:**
- Integrating eBPF (via `bcc` or `bpftrace`) requires kernel headers, elevated privileges, and platform-specific compilation. On Windows or macOS hosts, it is simply unavailable without a VM layer.
- Live `psutil` polling introduces timing dependencies: the ML model needs a minimum corpus of observations before the anomaly threshold stabilizes. In a 60-minute window, seeding the model with live data guarantees an under-populated training set for the first several minutes.
- Synthetic data allows Team 2 to inject controlled anomaly signatures (CPU spike to 98%, memory leak pattern, sudden zero-network-activity drop) with 100% reproducibility, making the demo deterministic and compelling regardless of host machine state.

**Future Path:** Replace `generate_process_data()` with a `psutil`-backed streaming collector wrapped in a 30-second rolling window buffer.

---

### 2.2 Volatile Pandas DataFrames Instead of Persistent Storage

**Decision:** All telemetry and anomaly records live exclusively in two in-memory Pandas DataFrames (`df_all`, `df_anomalies`). No SQLite, no PostgreSQL, no Parquet files on disk.

**Justification:**
- Database provisioning (schema creation, connection string management, migration files) consumes 10–15 minutes of a sprint that has only 35 minutes of coding time.
- SQLite introduces a shared file path that must be coordinated across all three teams — a merge conflict vector eliminated entirely by keeping data in Python objects passed as function arguments.
- Pandas DataFrames are the native format for Scikit-Learn's `IsolationForest`, eliminating any serialization/deserialization step between Team 2's ML layer and the rest of the pipeline.
- Session-scoped data is sufficient for a monitoring dashboard: the user sees the current run's anomalies. Historical persistence is a post-sprint feature.

**Future Path:** Wrap the DataFrames in a `DataStore` class that flushes to SQLite on a background thread after the sprint.

---

### 2.3 Monolithic `app.py` Instead of Decoupled Microservices

**Decision:** All three functional domains — UI rendering, data generation + ML inference, and LLM bridging — are implemented as Python functions within a single file (`app.py`) orchestrated by Streamlit's execution model.

**Justification:**
- Microservice boundaries require inter-process communication contracts (REST, gRPC, message queues), Docker Compose, and port conflict resolution. Each of these is a 20-minute task in isolation.
- A monolith eliminates network-layer debugging during assembly. When the LLM bridge fails, the stack trace is in the same file as the UI code — no distributed tracing required.
- Streamlit's top-to-bottom re-execution model is itself a form of micro-orchestration: state flows via `st.session_state` and function return values, which mirrors the microservice data contract without the infrastructure overhead.
- The assembly protocol (paste three function blocks into one file) is executable in under 5 minutes by a single person at the 45-minute mark.

**Future Path:** Extract each team's block into its own module (`data_engine.py`, `ml_engine.py`, `llm_bridge.py`) and import them into `app.py` as a first refactor step.

## 3. Tech Stack & Architectural Justifications

### 3.1 Streamlit (Frontend)

| Attribute | Detail |
|---|---|
| Version Target | `>=1.32.0` |
| Role | UI orchestrator, state manager, layout engine |
| Import | `import streamlit as st` |

**Why Streamlit:**
Streamlit converts Python functions directly into interactive web UI components with zero HTML, zero JavaScript, and zero CSS. For a sprint where all developers are Python practitioners, this eliminates an entire skill boundary. Its widget system (`st.button`, `st.dataframe`, `st.metric`, `st.spinner`) maps 1:1 to the UI elements Sentinel requires. The re-run execution model means any state change automatically refreshes all dependent visualizations — there is no event loop to manage manually.

Critically, Streamlit's `st.session_state` dictionary provides a lightweight shared memory store that persists DataFrames across button clicks, allowing Team 1 to hold `df_all` and `df_anomalies` without re-generating them on every render cycle.

---

### 3.2 Scikit-Learn Isolation Forest (ML Anomaly Engine)

| Attribute | Detail |
|---|---|
| Version Target | `>=1.4.0` |
| Class | `sklearn.ensemble.IsolationForest` |
| Mode | Unsupervised outlier detection |

**Why Isolation Forest:**
Isolation Forest is the only mainstream anomaly detection algorithm that requires no labeled training data, no GPU, no epochs, and no hyperparameter grid search to produce useful results on tabular telemetry data. It works by randomly partitioning the feature space: anomalous points (those that are statistically isolated) require fewer splits to isolate than normal points, resulting in a lower anomaly score.

For a system monitor with features like `[cpu_percent, memory_mb, net_bytes_sent, disk_read_ops]`, Isolation Forest is provably effective out of the box with `contamination=0.05` (flag the top 5% most isolated points as anomalies). It fits on a 500-row DataFrame in under 50ms on any modern CPU, adding zero latency to the Streamlit re-run cycle. No VRAM is consumed — this runs entirely on CPU, leaving the RTX 4050's 6GB free for Ollama's LLM inference.

---

### 3.3 Ollama (Local LLM Engine)

| Attribute | Detail |
|---|---|
| Version Target | `>=0.1.32` |
| API Endpoint | `http://localhost:11434/api/generate` |
| Protocol | Synchronous HTTP POST (JSON) |

**Why Ollama:**
Ollama provides a Docker-like CLI interface for pulling, running, and serving open-weight language models with automatic GGUF quantization and VRAM-aware layer offloading. It exposes a stable REST API at `localhost:11434` that accepts a JSON payload and returns a streaming or batched completion — no Python SDK required, no authentication, no rate limits.

For a 60-minute sprint, Ollama's value is in its operational simplicity: `ollama pull gemma2:2b` downloads and configures the entire model in one command, and `ollama serve` (auto-started on install) keeps it running as a background daemon accessible to any process on the machine. The HTTP interface means Team 3's LLM bridge requires nothing beyond the Python standard library's `requests` module.

---

### 3.4 Gemma 2B / Gemma 4B Edge (LLM Selection)

| Model | VRAM Usage | Quantization | Use Case |
|---|---|---|---|
| `gemma2:2b` | ~2.0 GB | Q4_K_M | Primary: fits within strict 6GB cap with headroom |
| `gemma2:4b` (optional) | ~3.5 GB | Q4_K_M | Upgrade path if workstation has ≥8GB VRAM |

**Why Gemma 2B:**
The RTX 4050 has 6GB of GDDR6 VRAM. With OS and display drivers consuming ~0.5GB, the effective available VRAM budget for an LLM is approximately 5.5GB. Gemma 2B in Q4_K_M quantization occupies ~2.0GB, leaving ~3.5GB free for parallel compute tasks (browser, IDE, other processes) during the sprint.

More importantly, Gemma 2B has sufficient instruction-following capability to perform structured prompt-to-narrative translation — which is all Sentinel requires. The prompt format is tightly constrained (a small JSON-like anomaly summary), and the expected output is a 3–5 sentence security alert. Gemma 2B handles this reliably without hallucinating outside the provided context.

Gemma 4B is offered as an optional upgrade for workstations with more VRAM, providing measurably more coherent narratives for complex multi-process anomaly clusters.

---

### 3.5 Requests (HTTP Bridge Library)

| Attribute | Detail |
|---|---|
| Version Target | `>=2.31.0` |
| Usage | Synchronous POST to Ollama REST API |
| Import | `import requests` |

**Why Requests:**
The `requests` library is the de facto standard for synchronous HTTP in Python. It requires no async event loop, no connection pool configuration, and no protocol negotiation. For a single synchronous call to `localhost:11434`, it is the lowest-ceremony option available. Using `httpx`, `aiohttp`, or `urllib3` directly would introduce either async complexity or boilerplate that provides zero functional benefit within the sprint scope.

## 4. Functional Specifications

### 4.1 Data Contract: `df_all`

The primary telemetry DataFrame produced by Team 2 and consumed by Team 1 (for display) and the ML model (for scoring).

| Column | Type | Unit | Description |
|---|---|---|---|
| `timestamp` | `datetime64[ns]` | — | UTC timestamp of the simulated observation |
| `process_name` | `str` | — | Simulated process name (e.g., `chrome.exe`, `python.exe`) |
| `pid` | `int64` | — | Simulated process ID (100–9999) |
| `cpu_percent` | `float64` | % | CPU utilization at observation time (0.0–100.0) |
| `memory_mb` | `float64` | MB | Resident set size in megabytes |
| `net_bytes_sent` | `float64` | bytes | Network bytes sent in the observation window |
| `disk_read_ops` | `float64` | ops/s | Disk read operations per second |
| `anomaly_score` | `float64` | — | Raw Isolation Forest score (negative = more anomalous) |
| `is_anomaly` | `bool` | — | `True` if `anomaly_score` is in the contamination tail |

**Row Count:** 500 rows per generation cycle (configurable via `N_SAMPLES` constant).

---

### 4.2 Data Contract: `df_anomalies`

A filtered subset of `df_all` containing only rows where `is_anomaly == True`. Produced by Team 2, consumed by Team 1 (for display) and Team 3 (for LLM prompt construction).

| Column | Type | Notes |
|---|---|---|
| All columns from `df_all` | — | Same schema, filtered subset |

**Row Count:** ~25 rows (5% contamination of 500 rows). May vary due to random seed.

---

### 4.3 Function Contract: Team 1 — `render_dashboard(df_all, df_anomalies)`

```python
def render_dashboard(df_all: pd.DataFrame, df_anomalies: pd.DataFrame) -> None:
    """
    Renders the Streamlit UI.
    - Displays summary metrics (total processes, anomaly count, anomaly rate).
    - Renders df_all in a scrollable table with anomaly rows highlighted.
    - Renders df_anomalies in a separate section.
    - Provides a button that triggers get_llm_explanation(df_anomalies).
    - Displays the returned LLM explanation string in st.info() or st.markdown().
    """
```

**Input:** `df_all: pd.DataFrame`, `df_anomalies: pd.DataFrame`
**Output:** `None` (side effects: Streamlit renders)
**Dependency:** Calls `get_llm_explanation()` from Team 3 on button click.

---

### 4.4 Function Contract: Team 2 — `generate_process_data()` and `run_anomaly_detection(df)`

```python
def generate_process_data(n_samples: int = 500) -> pd.DataFrame:
    """
    Generates a synthetic process telemetry DataFrame with n_samples rows.
    Injects anomaly signatures into approximately 5% of rows.
    Returns a DataFrame matching the df_all schema (without anomaly_score and is_anomaly).
    """

def run_anomaly_detection(df: pd.DataFrame) -> tuple[pd.DataFrame, pd.DataFrame]:
    """
    Accepts the raw telemetry DataFrame.
    Fits an IsolationForest on numeric feature columns.
    Appends anomaly_score and is_anomaly columns to the DataFrame.
    Returns (df_all, df_anomalies) as a tuple.
    """
```

**Input:** `n_samples: int`
**Output:** `tuple[pd.DataFrame, pd.DataFrame]` — `(df_all, df_anomalies)`

---

### 4.5 Function Contract: Team 3 — `get_llm_explanation(df_anomalies)`

```python
def get_llm_explanation(df_anomalies: pd.DataFrame) -> str:
    """
    Accepts the anomalies DataFrame.
    Extracts the top 5 anomalous rows by anomaly_score.
    Constructs a structured security analysis prompt.
    Sends a synchronous POST request to http://localhost:11434/api/generate.
    Parses the response JSON and returns the LLM's explanation as a plain string.
    Returns an error string (not an exception) if Ollama is unreachable.
    """
```

**Input:** `df_anomalies: pd.DataFrame`
**Output:** `str` — human-readable security narrative or error message
**Side Effects:** One blocking HTTP POST to `localhost:11434`

---

### 4.6 Orchestration Entry Point

```python
# app.py — main() is the Streamlit entry point
def main():
    st.title("Project Sentinel — Local Anomaly Monitor")
    df_raw = generate_process_data(n_samples=500)
    df_all, df_anomalies = run_anomaly_detection(df_raw)
    render_dashboard(df_all, df_anomalies)

if __name__ == "__main__":
    main()
```

## 5. Non-Functional Requirements

| Requirement | Target | Notes |
|---|---|---|
| Startup time | < 5 seconds | `streamlit run app.py` to first render |
| ML fit + predict latency | < 500ms | On 500-row DataFrame, CPU only |
| LLM response latency | 10–45 seconds | Gemma 2B on RTX 4050, varies by prompt length |
| VRAM ceiling | < 3.0 GB | Must not evict OS or display memory |
| External network calls | 0 | All traffic stays on localhost |
| Disk writes | 0 | No files written during operation |
| Python version | >= 3.10 | Required for `tuple[...]` type hint syntax |

## 6. Out of Scope (Sprint 1)

- Live kernel telemetry via eBPF, `psutil`, or WMI
- Persistent storage (SQLite, PostgreSQL, Parquet, CSV export)
- User authentication or multi-user sessions
- Anomaly alert notifications (email, Slack, PagerDuty)
- Model retraining on accumulated historical data
- Containerization (Docker, Podman)
- Unit tests, integration tests, CI/CD pipelines
- Windows Service or systemd daemon packaging
